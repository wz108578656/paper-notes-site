    论文分析报告 通俗讲解¶

    0. 整体创新点通俗解读¶

    好，我们直接进入正题。作为你的导师，我帮你把这几十页的内容扒开，看看它的“灵魂”到底是什么。

    核心问题：现在的CPU和内存之间存在一个巨大的鸿沟。CPU速度快得像火箭，内存却慢得像牛车。 过去大家怎么解决？ - 硬件级投机：让硬件自己去猜你接下来要读什么（Prefetcher）。但这就像蒙着眼睛猜谜语，猜对了固然好，猜错了不仅浪费带宽，还会把宝贵的缓存搞乱（Cache Pollution）。 - 硬件级调度：在内存控制器里，通过复杂的排队算法（FR-FCFS、PARBS等）来试图管理不确定性。但这依然是“瞎子摸象”，它看不到程序整体的访问意图。

    最难受的地方：当多个程序（尤其是流式应用，如视频处理、线性代数）同时访问内存时，硬件就像个混乱的十字路口，谁都想先走，结果谁也走不快，还经常发生“车祸”（Row Conflict）。这种由不确定性带来的性能损失和延迟抖动，对于追求绝对安全的高性能或实时系统来说，是无法接受的。

    想象一下，你是一个大型演唱会（现代计算机系统）的后台总管（Memory Controller）。 - 传统做法：你手下有几十个跑腿的（CPU Core/硬件Prefetcher）。他们是艺术家（Software）的助理。艺术家们需要什么道具（Data），就让助理跑到后台仓库（DRAM）去拿。助理不知道仓库怎么规划的，只知道艺术家喊什么，他就冲进去乱翻。结果几十个助理挤在仓库门口，互相踩脚，把仓库弄得乱七八糟。有的助理（Prefetcher）还自作聪明，觉得艺术家可能需要某样道具，提前冲进去拿，结果发现拿错了，平白增加了拥堵。

    • 论文的做法（InterStellar/COMPASS）：论文提出，不要把事情全交给这些乱跑乱撞的跑腿助理了。我们要让艺术家（软件） 直接跟我们后台总管（硬件） 沟通。 艺术家在上台前，就给我一个 “演出流程单” (Stream Descriptor)：“我第一幕需要红色沙发，第二幕需要蓝色桌子，第三幕需要那面大鼓。”

    于是，作为后台总管，我就能做两件很聪明的事： 1. 提前备货：我让仓库管理员在艺术家需要的前一秒，提前把“红色沙发”搬到门口（Intelligent Batching / iBatch），艺术家一挥手，东西就到了。 2. 仓库规划：我知道红色沙发在仓库A区，蓝色桌子在B区。我不会让两个助理在A区和B区之间来回跑，导致仓库瘫痪。我会把A区的东西一次性处理完（Intelligent Page Policy / iPP），再转向B区，避免冲突。

    这帮作者没有去重造一个更快的仓库（DRAM），也没有训练更聪明的跑腿助理（Prefecther）。他们做了一件看似简单但极其巧妙的事：

    在“艺术家（软件）”和“仓库总管（硬件）”之间，建立了一条直通的信息专线。

    具体逻辑转换是： - 原本流程：软件发出随机请求 -> 硬件“盲猜” -> 内存执行。 - 论文流程：软件“提前告诉我”未来的访问模式 -> 硬件“根据剧本”主动且精确地获取并批量调度请求 -> 内存高效执行。

    这条专线就是文中的 “HW/SW Co-design Interface”。软件通过编译器等手段，把数据流的“剧本”（Base Address, Stride, Loop Bounds）编译成极简的指令包（Descriptors），通过现有寄存器（RISC-V CSR）直接告诉硬件。

    硬件（Nucleus）在拿到剧本后，就从一个“被动执行者”变成了一个“主动管理者”。它不再是等请求来了才开始干活，而是提前规划： - 在内存控制器层面，它把属于同一行数据的未来请求打包成一个rtBatch，一次性发射，让数据像军队一样整齐划一地流动，彻底避免了“十字路口的混乱”。 - 在缓存层面，它的智能预取器（iPF）只取剧本里有的数据，绝不乱猜，并且和iMC协同，确保前端预取和后端调度节奏一致，没有内耗。

    核心洞察：这篇论文最大的贡献不是某个具体的硬件电路设计，而是一个系统性的哲学转变：从“硬件单方面猜”转变为“软硬件双向沟通”。它证明了，在一个高度受限的系统中，“让软件多说一句话”比“让硬件多猜一万次”要高效和可靠得多。

    1. 它解决了什么问题？ 内存系统中，硬件“盲猜”导致的性能差和不可预测性。
    2. 它提出的核心方法是什么？ 让软件主动告知硬件未来的精确内存访问模式（流式描述）。
    3. 它带来的直接好处是什么？ 硬件（内存控制器和缓存）可以根据“剧透”进行主动、批量、无冲突的预取和调度，从而大幅提升性能和可预测性。

    这就是这篇论文的“魂”。看懂了这个，里面的所有技术细节都只是这个灵魂在不同硬件层次上的具体实现形式。

    1. HW/SW Interface and Stream Descriptors¶

    之前的内存控制器就像一个盲人快递员：它只知道当前手里这一单的地址（物理地址），对于接下来还有多少包裹、是继续往东走还是突然掉头，一概不知。
    - 传统硬件事先无法区分“这是一个大数组的连续访问”还是“随机跳转”，因此只能用通用的开放页/关闭页策略去猜。猜对了就赚，猜错了就付出行冲突的高昂代价。
    - 即使有硬件预取器，它们也是从已发生的访问模式里事后学习，永远慢半拍，而且容易把不同数据结构的流搅在一起，自相干扰。
    - 最关键的是：软件明明在循环体里就清清楚楚知道下一步要访问哪，但硬件却听不见——这种“语义鸿沟”导致大量DRAM带宽浪费在无意义的预充电和激活上。

    想象你在一家智能餐厅后厨负责备菜。
    - 传统做法：厨师（CPU）不断喊“来一份宫保鸡丁的鸡肉”“来一份宫保鸡丁的花生”——你（Memory Controller）每次听到一个订单，才跑去冰箱找对应食材。因为厨师喊得忽快忽慢、食材还分冷藏区和冷冻区，你每次开门都要花时间，手忙脚乱。
    - HW/SW Interface的思路是：厨师在开始炒这道菜之前，直接给你一张清单：“接下来30秒内，我只需要：鸡肉×5，花生×3，干辣椒×2，都在同一个冷藏格里。” 你拿到清单，一次性把冷藏格打开，把清单上的东西全取出来放工作台上。后面厨师每喊一次，你直接从工作台递过去，再也不用反复开关冰箱门。

    这张清单就是“Stream Descriptor”——它把软件的意图压缩成几个数字，提前告诉硬件。

    作者没有修改指令集或操作系统，也没有在运行时动态分析访问模式，而是让编译器在生成代码前就把流的信息写好。

    具体做法：
    - 编译器在LLVM pass里扫描循环中的数组访问，识别出那些基地址+固定步长+确定循环次数的访问模式，把它们打包成几个紧凑的描述符。
    - 循环描述符：记录该循环的起始、终止、步长（比如 for i=0; i<100; i+=1）。
    - 直接流描述符：记录基地址和步长（比如数组A，步长8字节）。
    - 间接流描述符：记录通过另一个数组间接访问的模式。
    - 链接变量描述符：当基地址或边界在运行时才确定时，记录从哪里动态读取。

    • 这些描述符通过标准的CSR写指令（RISC-V的csrrw）在循环进入之前就写入硬件寄存器。不需要新指令，不需要OS参与。

    • 硬件端的Nucleus引擎在收到每个Last-Level Cache Miss时，利用一个反向翻译表（Stream TLB^-1）把物理地址转回虚拟地址，然后与描述符表中的条目比对。一旦匹配上，就在这个内存请求上附上流ID、流类型（直接/间接）、步长等信息，再发给Memory Controller。

    • Memory Controller拿到带标签的请求后，就不再是“睁眼瞎”了：它知道这个请求属于哪个流，下一步还有多少个兄弟请求，于是可以决定是否批量预取、是否保持当前行打开，以及跨通道时如何切分batch。

    本质上，这个接口把编译时已知的访问拓扑编译成若干个几十比特的描述符，用几条CSR写操作注入硬件，从而在零额外运行时开销的前提下，让硬件获得了“预知未来”的能力。

    2. Nucleus Engine¶

    传统内存控制器就像一个聋子调度员——它不知道哪个程序（core）发来的请求，更不知道这些请求接下来要去哪里、有什么规律。结果就是： - 流式数据被拦腰斩断：明明是一个顺序读写的大循环，由于缺乏上下文，控制器把连续的请求当成孤立事务处理，导致频繁的row冲突和额外的precharge/activate延迟。 - 调度撞墙：多个核心的流相互交织，控制器只能基于当前队列里的随机顺序做决策（FR-FCFS），效率极低且无法预测。 - 协调真空：缓存和内存控制器之间没有语义通道，预取器猜测的请求可能污染cache，而控制器却无法利用这些“即将到来”的信息提前优化。

    这些问题在AI/ML和科学计算场景下尤其致命，因为这些负载的核心就是重复遍历大型数组。

    想象一个繁忙的图书馆前台：读者（cores）不断递来索书条（memory requests）。前台小姐姐（传统memory controller）只能看纸条上的书名号（physical address），然后机械地跑到对应书架（DRAM bank）去取书。她完全不知道某个读者接下来还会要什么书，也不知道另一个读者是否也在借同一本书。

    现在，图书馆引入了一个智能助手（Nucleus Engine）。这个助手在读者进门时就拿到了他们的研读清单（stream descriptors）——上面清楚写着：“我要从《机器学习》第10页开始，每隔10页读一页，共读30页。” 助手提前把这份清单翻译成书架坐标，并告诉前台：“这位读者接下来的5次请求都要去同一个书架，你先别急着关书架门；等看完这5本，你再自动把门关上。” 这样一来，前台不再盲目狂奔，而是精准地批量操作。

    作者没有对内存控制器动大手术，而是在LLC边界插入了一个轻量级硬件助手（Nucleus）。它的核心操作是“身份识别 + 提前翻译 + 打标签”：

    1. 解析软件清单：编译器在循环开始前，把循环的步长、边界、访问模式编码成紧凑的描述符（descriptors），通过CSR写入。Nucleus读取并存储这些信息，计算出每条流的虚拟地址范围。
    2. 解决虚实地址映射：由于LLC看到的是物理地址，而描述符是虚拟地址，Nucleus维护了一个极小的反向TLB（TLB⁻¹），当TLB发生miss时，自动记录物理页到虚拟页的映射。这样就能把传入的物理地址快速归类到对应的流。
    3. 打标签并附加：每个LLC miss在离开LLC之前，Nucleus会判断它是否属于已知流，然后在请求包上附加三个关键标签：Stream ID（属于哪条流）、Core ID（哪个核心发起）、Access Type（直接流/间接流/非流）。这些标签随请求一起前往内存控制器，后者根据这些信息执行智能的页面策略和批量预取。

    杠杆点：这个设计把软件层面的循环结构知识，无缝转换为硬件层面的元数据，且整个过程对cache hit路径完全透明（仅在miss时多两个周期，且与LLC tag查找并行），代价极低。从此，内存控制器不再是瞎子，而是带薪看股市行情——它能提前知道未来要读什么，从而把DRAM的row buffer利用率提到最高。

    3. Intelligent Page Policy (iPP)¶

    好的，让我们来拆解这个Intelligent Page Policy (iPP)。它听起来像是个总策略，但你给的线索很清晰，它其实是根据流 (Stream) 的不同脾气，采取两套完全不同的治理方法。

    1. 痛点直击

    传统的DRAM Page Policy，比如Open-Page或Close-Page，都是一刀切的。它们不知道下一个请求是来自一个规律的Vector循环，还是一个随机的Graph搜索。就像管理员对所有房间都用同一种方式管理：要么门全开着（Open-Page），等你走了再关，但可能被人闯入放东西而变慢；要么门随时关着（Close-Page），安全但每次进出都得重新开门。

    这就导致了在混合访问模式下，系统很难受。当一个规律的大批量请求（Direct Stream）和时不时过来蹭一下的随机请求（Indirect Stream）混在一起时，传统Policy会顾此失彼。Open-Page会让随机的冲突导致巨大的切换开销；Close-Page则会让规律流浪费掉珍贵的Row Hit红利。

    2. 通俗比方

    你可以把DRAM的Row Buffer想象成一个图书馆的临时桌面。 - Direct Stream 就像图书馆员今天要处理一整套百科全书（规律的大阵列）。他需要一本接一本地翻阅，看完接着看下一本。 - Indirect Stream 则像一个临时来查资料的读者，他可能东翻一本，西翻一本（随机访问），没有固定顺序。

    传统的管理员（Page Policy）是糊涂的。他可能因为那个临时读者翻了一本，就把这一摞百科全书对应的桌面清空了（Precharge），结果导致图书馆员再回来时，不得不重新从书架搬书（冲突），效率极低。或者他全程开着桌面的书，让临时读者在那一排书里乱翻，导致冲突。

    iPP这个新管理员的聪明之处在于：他能看穿图书馆员（Direct Stream）的工作单（Stream Descriptor），知道“哦，你刚翻完了第1册，接下来还有第2、3、4册要连续看”。因此，他会决定先别清空桌面，让图书馆员赶紧把剩下的几本看完（利用Row Hit）。同时，他也认得那个临时读者（Indirect Stream）。如果这个读者反复换不同主题的书（频繁冲突），他就能做出判断：“这家伙明显在瞎逛，与其让他干扰别人，不如我先把他那份桌面清空（Precharge），让他下次自己重新搬书，别拖累图书馆员。”

    3. 关键一招

    iPP没有试图去“预测”未来，而是引入了“为不同的流建立独立的感知模型” 这一逻辑转换。它把传统的、一视同仁的Bank级决策，替换为了Stream级决策。

    具体来说，它引入了两个“小本本”： - Row-Hit Counter (针对 Direct Stream)：这不是什么新奇的硬件，但作者巧妙地用软件传过来的 Stride 算出了 MaxHit。这个MaxHit就是“在当前这页Row Buffer里，你能连续读到多少行数据”。当这个计数器达到上限时，iPP就知道：“行了，这绝大部份的潜力已经榨干了，该准备关门（Precharge）了。” 这本质上是一个根据固定速率计算出的、精确的“提前关门”信号，避免了Row被长时间占用而引发冲突。 - Conflict-Tracking Window (针对 Indirect Stream)：对于无规律的访问，iPP建了一个滑动冲突计数器。当冲突次数超过一个高阈值时，就强行关门；当冲突次数落在一个低阈值上下且连续冲突时，也预期性地关门。这相当于一个基于行为观察的自适应“紧急刹车”机制，能在冲突真正恶化前，提前给所有流让出通道。

    总结： iPP的核心进步不在于更快的硬件，而在于从“反应式”的Bank粒度管理，进化到了“前瞻式”的Stream粒度管理。它不再是一刀切，而是针对“规律的”和“随机的”请求分别制定了最优策略。

    4. Intelligent Batching (iBatch)¶

    痛点直击

    • 传统的硬件流预取器（Stream Prefetcher）虽然能“猜”到接下来的地址，但它们是从 Cache 层级 发起的。这就导致了两大问题：
    • 请求到达 DRAM 时已经“乱序”：预取请求和需求请求在内存控制器队列里混杂，导致频繁的行冲突（Row Conflict）。每次切换行都要重新激活，白白浪费几十个周期。

    • 资源争抢严重：预取请求会挤占 MSHR、Cache 容量和片上互联带宽。很多时候预取的还没用上，就已经把有用的数据挤出去了。

    • 最终表现是：DRAM 的实际本地化（Row Hit）率依然很低，大部分时间都花在激活/预充电上，有效带宽被严重浪费。

    通俗比方

    • 这就像机场的 行李转盘：传统做法是每个乘客（需求请求）走到转盘边，然后地勤根据乘客的票根（地址）去后面仓库（DRAM 阵列）搬箱子。由于事先不知道下一个乘客是谁，地勤每搬一个箱子就要把转盘转回到起点，再搬下一个。

    • 另一种做法是给地勤一张 航班行李清单（Stream 描述符）。当第一个乘客站到转盘前时，地勤不仅把他的箱子放上去，还提前把所有同航班的箱子都一次性放上传送带。后面的乘客不用等，直接取走就行。

    • iBatch 做的正是类似的事：它不在“登机口”（Cache）就提前把箱子拿出来占地儿，而是在 转盘控制室（Memory Controller） 里看着，等到第一个乘客站定（Row Open），才一口气把该航班所有剩下的箱子（本行的所有 Cache Line）都搬到转盘上。

    关键一招

    • 作者并没有在 Cache 和 DRAM 之间增加复杂的协调逻辑，而是做了一次 “管辖权转移”：
    • 发起点 从 Cache 层（预取器）移到 内存控制器层。
    • 触发条件 从不确定的时间（预取器随机发起）变为 当一个需求请求打开某一行后。
    • 利用软件提供的 Stride 和 Stream 长度，控制器在该行打开后，立即计算出该行剩余的所有地址（这些地址几乎 100% 是 Row Hit），并以一个不可中断的 Burst 一次性发出全部读取请求。
    • 返回的数据暂存在每个 Bank 的小型 iBuffer 中。后续需求直接命中 iBuffer，无需再访问 DRAM 行缓冲。

    • 与 iPP（智能页面策略） 配合：当 Batch 结束，iPP 自动预充电关闭该行，保证下一个 Stream 的第一个请求也不是冲突（而是 Miss，代价更小）。

    • 简单说：不是提前猜，而是确定了再批量取。 而且这个“批量”的时机和大小都由软件的 Stream 描述和控制器的 Row 状态共同决定，所以做起来确定性很强，不会浪费带宽。

    5. Real-Time Batching (rtBatch) and BR-FR-FCFS Scheduling¶

    • 在多核实时系统中，多个核共享DRAM，传统的First-Ready First-Come-First-Serve (FR-FCFS) 调度器虽然优先处理行命中请求，但遇到流式访问（affine stream）时，情况依然糟糕。
    • 假设一个核正在按顺序读数组A，每隔几十个cycle到达一个Read命令。这些命令虽然命中同一行，但问题在于，它们不是同时到达的。当第一个Read被执行后，第二个Read需要等待，此时其他核的请求（比如写操作、其他行的读）可能插进来，导致行冲突或Bank Group干扰。最终，即便原本都是行命中，最差情况延迟（WCML）依然非常高，因为无法保证这批请求能够连续、无中断地被服务。
    • 之前的方案（如Bank分区）能减少跨核行冲突，但解决不了同一核内因请求到达时间分散而引入的干扰窗口。WCML依然被无关的Write Batch或跨Bank Group的时序约束拉长。

    想象你是一个快递员，需要给同一个小区（同一DRAM行）的10户人家送包裹。

    • 传统做法：你回站点取一个包裹，跑一趟送掉；再回站点取第二个，再跑一趟。每次来回，路上都可能被其他快递员（其他核的请求）挡道、抢电梯，甚至被临时通知去送别的小区（行冲突）。最坏情况下，这10趟送完可能要花很久，时间完全不可预测。
    • rtBatch做法：你一次性从站点取走这10个包裹，然后一口气进小区，挨家挨户敲门送完，中间不回站点、不理其他任务。这样，送这10个包裹的时间就是确定且紧凑的：走路进小区的时间（ACT）加上每户敲门送件的时间（CAS across BGs）。即使隔壁小区也有快递员，他也插不进你这趟“专线”。

    作者并没有改变DRAM的基本时序，而是从“请求的到达方式”和“调度秩序”两方面入手：

    1. rtBatch（实时批生成）：利用软件提供的流描述符（stride、base地址），一旦第一个读请求（demand）被调度，内存控制器立即预生成接下来同一行的B-1个读请求，并把这B个请求捆绑成一个原子批。这批请求有相同的优先级，且全部命中当前行。

    2. BR-FR-FCFS调度（批就绪优先）：调度器增加一条规则——如果某个批已就绪（即所有段的第一个请求都满足时序），就优先执行整批，不允许中断。具体到Bank Group (BG) 交错的地址映射（Map2），这批请求被安排到不同BG，因此批内请求的间隔是最短的tCCDS（4 cycle），而不是同BG的tCCDL（6 cycle）或更长的行冲突延迟。调度器保证批的两个段（对应两个BG）是顺序执行的，不会因其他核的请求而暂停。

    3. 自适应页策略：每个rtBatch结尾自动使用带预充电的读命令（RDA），提前关闭行，使得下一个rtBatch的第一次访问必然产生行未命中（状态清晰），避免了随机冲突带来的不可预测性。这样，批内延迟固定（tRCD + (B-1)tCCDS），批间等待时间也可用公式精确上界*（取决于tRC和tRTP）。

    核心逻辑转换：不再让每个读请求在队列里“公平竞争”，而是让已知连续且可预测的流式请求武装成一个“特权车队”，获得一条无干扰的专用车道。这牺牲了一点公平性，但换来了严格可计算的最差情况延迟——实时系统最需要的恰恰是这种确定性。

    6. Coordinated Prefetcher and Memory Controller (COMPASS)¶

    • 之前的做法：缓存预取器 (iPF) 和内存控制器 (iMC) 各管各的，存在严重的信息孤岛。
    • 预取器基于过去模式发出大量预取请求，但它完全不知道内存控制器此刻正在批处理 (Batching) 哪一组行命中请求。导致预取请求往往打散了已形成的连续批，迫使DRAM不断换行 (Row Conflict)。
    • 内存控制器做的批处理（如PARBS/BLISS）只看当前队列里的请求，但它看不到未来一小段即将到达的、必定是命中当前行的预取数据。结果，批处理窗口要么太小浪费带宽，要么太大吃掉本可以等到的预取。
    • 核心崩溃场景：一个预取器刚刚把下一行预取到LLC，紧接着内存控制器就因为当前批处理结束而关闭了开着的行 (Auto-Precharge)。预取的数据虽然进了LLC，但DRAM行已经关了，后续需求过来时，又得重新激活 (ACTIVE)，白白浪费了两个周期的时间。
    • “补丁式”改进的极限：之前的工作要么只优化预取器（尽量踩点），要么只优化调度器（尽量等待），但它们没有“握手”，永远存在时序错位 (Timing Mismatch)。

    想象一个日料流水席：

    • 预取器是一个厨师，他根据菜单提前准备寿司（预取数据）。
    • 内存控制器是传菜员，他每批只能端4盘（批处理），并且一次只能从一个灶台（DRAM行）拿菜。
    • 原来的问题：厨师（预取器）凭感觉预制寿司，可能同时在做三盘不同灶台的菜。传菜员（内存控制器）看到哪个灶台菜多就先端，结果两种菜轮流端，导致每个灶台都要重新开火（ACTIVATE）和关火（PRECHARGE），效率极低。更惨的是，厨师做的寿司可能传菜员根本来不及端（LLC污染），或者传菜员端完一批后，厨师才把下一批寿司做好（定时错位）。

    COMPASS的解决方案：在厨师和传菜员之间加一个调度看板（iCoordinator）。

    • 这个看板上写着：“接下来5秒，只做三文鱼（某个流/某个行），做成16盘，分4批端”。
    • 厨师（预取器）只看这个看板，只做三文鱼，且每做成4盘就通知传菜员。
    • 传菜员（内存控制器）只看这个看板，只端三文鱼，并且保证每次去端时，灶台上都刚好有4盘新鲜的寿司等待。

    这样一来，灶台（DRAM行）始终是热的，厨师不用乱做菜，传菜员不用空手跑。整个流水席的吞吐量和响应时间都得到了保证。

    作者并没有发明一种全新的调度算法，也没重设计预取器，而是巧妙地、以极低成本，在LLC和内存控制器之间插入了一个“对齐层”（iCoordinator）。

    • 思想转变：从 “独立猜测+被动调度” 变为 “基于契约的协同”。
    • 原来，预取器猜内存后面会缺，然后发个请求过去，至于内存控制器怎么排，它不管。
    • 现在，iCoordinator基于软件提供的stream desc，提前告诉iPF和iMC：接下来一段时间，这个流只会访问这个DRAM行，并且预取和批处理的步调必须一致。
    • 微架构具体做法：
    • iPF不再盲目发单点预取。它收到指令后，批量地、按顺序地向LLC填充数据。
    • iMC不再自己猜该批多大。它从iCoordinator获取预取完成的时间戳，从而智能地扩大或缩小批处理窗口，确保批处理的第一个请求命中行，最后一个请求也是行命中，中间不断开。
    • 同步点：iCoordinator设置一个“批启动信号”。只有当iPF确认下一批预取数据已就位（或即将就位）时，iMC才启动该批。这消除了超时等待和错误启动。
    • 核心胜利：实验显示，这种对齐机制在标准配置下，将平均端到端加速比从预取器最高的1.63x（仅iPF）提升到1.88x，而仅仅使用预取和调度器组合的1+1>2效果，关键在于对齐。

    一句话总结：COMPASS没有增加更多计算单元，而是让两个原本“各扫门前雪”的组件学会了“看同一块表”指挥---共享时钟源和任务序列表。
