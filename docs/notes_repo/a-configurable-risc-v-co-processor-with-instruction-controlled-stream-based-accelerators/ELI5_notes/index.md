# A Configurable RISC-V Co-Processor with Instruction-Controlled Stream-Based Accelerators 通俗讲解

### 0. 整体创新点通俗解读

- **之前做法的困境**：要在RISC-V系统中集成多个硬件加速器，要么走**松散耦合**（通过总线/AXI连接），此时控制加速器需要开发者自己写底层驱动、手动管理数据搬运和共享内存，编程负担极重；要么走**紧密耦合**（如通过RISC-V的自定义指令或RoCC接口），但这往往只为特定计算任务（如向量、加密）设计，**不具备通用性和可扩展性**，无法灵活组合多个不同的流式加速器。
- **核心矛盾**：加速器需要**高带宽、低延迟**的数据访问，而通用核心与加速器之间数据搬运的**软件开销**和**时钟域差异**成了瓶颈。同时，多个加速器如果要**链式工作**（一个的输出是另一个的输入），传统做法必须经过内存中转（写回再读），浪费带宽和延迟。
- **论文要解决的大问题**：如何提供一个**既通用又高效**的协处理器平台，让开发者像调用软件函数一样（用指令）轻松控制多个硬件加速器，同时硬件上支持加速器之间直接数据流、独立时钟频率，且资源占用小、可配置性强。


- **Imagine a well-run factory**：核心是**厂长（RISC-V CPU）**，协处理器是**中层管理部门**。过去，厂长得亲自跑到每个工位（加速器）旁边，口述指令（写驱动/API），还得自己推着小车（数据搬运）来回跑，工位之间想传递半成品，必须先把半成品堆到仓库（内存），再推走，效率极低。
- **本文的做法**：厂长通过**一叠标准化指令卡片（自定义指令）**，下发给中层管理部门。管理部门内部有一个**自动化流水线作业区（Execution Unit）**，里面有很多**独立工位（Compute Units，CUs）**，每个工位连接一个专业机器（AXI-Stream加速器IP）。这些工位之间通过**传送带（Forwarding Unit，FWU）** 直接传递半成品，完全不用经过仓库。每个工位自己有小料斗（地址队列），可以按非连续的顺序取用原料。更重要的是，厂长自己的办公桌（核心时钟）和流水线作业区（协处理器时钟）**使用不同的时钟频率**，流水线可以跑得比厂长快得多，且互不干扰。


- **关键逻辑转换**：作者没有把加速器当作一个外设或函数单元，而是**抽象成“有输入输出流的微操作”**，并用**RISC-V自定义指令**统一管理其整个生命周期：配置、启动、等待完成、数据搬运。这相当于给每个加速器配了一个**标准的硬件控制器（CU Controller）**，将加速器暴露的AXI-Stream接口映射成**内存映射的读写操作**。
- **具体巧妙的替换**：
  - **替换数据搬运方式**：作者**没有**用传统的DMA或API来让核心搬运数据到加速器。而是通过**扩展数据桥（Data Bridge）**，将协处理器的scratchpad内存直接映射到RISC-V核心的地址空间。这样核心可以用普通的load/store指令直接读写加速器的本地数据区，移除了之前必须通过XIF接口进行专用指令搬运的低效和阻塞。
  - **替换IP间通信方式**：引入了**转发单元（FWU）**。它在互联网络中扮演一个“虚拟从设备”，**并不对应实际的内存**。当生产者CU向FWU的地址写数据时，消费者CU同时从FWU读数据，FWU在内部直接握手转发，**绕过了所有内存读写**。这一招让多个流式加速器实现硬件层面的**直接管道（Pipeline）**。
  - **解决非连续访问**：给CU增加了**地址队列（Address Queue）**。核心可以预先生成一个地址序列（类似索引数组）存到scratchpad，然后通过两个新指令（XLDA / XCFGA）告诉CU：“先去那个队列里把地址读进来，然后按这些地址去取数据喂给加速器”。这使流式加速器也能支持**不规则数据重组**，而不用核心逐条干预。
  - **时钟域解耦**：在协处理器内部，将**核心逻辑（指令解码、配置）**和**执行单元（互联网络、加速器、内存）**划分到两个独立时钟域，中间用**异步FIFO**缓冲。这样执行单元可以跑到更高频率（如333 MHz），核心只需跑在较低频率上，兼顾了能效与性能。

### 1. Custom Instruction Set Extension via XIF

好的，我们来聊聊**“通过 XIF 扩展的自定义指令集”**这个设计。简单来说，它解决的是一个“沟通成本”的问题。


想象一下，你是一个项目经理（RISC-V CPU），手下有几个干活特别快的专家小组（硬件加速器IP）。在之前，你要给小组布置任务，流程是这样的：

-   你得先走到一个**公共公告板**（内存映射寄存器或共享内存）上写通知：”任务A：数据在xxx地址，做完放到yyy地址”。
-   然后你冲专家小组大喊一声：“干活！”（触发一个外设中断或轮询标志位）。
-   小组干完活，再回到公告板上写“干完了”。
-   你又得跑过去把结果抄回来。

这个过程极其**烦琐、串行且低效**。CPU 绝大部分时间不是在思考（执行逻辑），而是在跑腿（处理数据移动和同步）。你每次布置任务，都要经历一套沉重的 **API 调用**，这里面包含了大量的**内存读取、写入和同步开销**。这就像两个公司协作，每次沟通都要写正式公函，而不是直接发个微信语音。


这个设计的精髓，就是**把“公函式沟通”变成了“指令式沟通”**。

**你的思维模型可以是：** 想象你不是在用公告板，而是给每位专家小组配了一个**对讲机**（XIF接口）。
这个对讲机只有一个频道，专门用来发**短命令**。比如：

-   你拿起对讲机说：“01号小组，开始任务Alpha，数据源：仓库A，结果放仓库B。”
-   对讲机那头回一个字：“收到。”
-   之后你不用管了，01小组会自己读取仓库A，处理完放仓库B。你同时可以继续给02小组发下一个指令。
-   当你想知道结果时，直接问：“01，完事了吗？” 对方可能回：“已完成”或者“请稍等”。

这个过程，**CPU 不再需要亲自“拿着钥匙去开仓库门”**。它只是**发布指令**，然后**等待结果**。XIF 就像是 RISC-V 处理器预留的一个“VIP通道”，专门用来和内部的特种部队（协处理器）进行高效、低延迟的通讯，而不是走拥挤的公共总线。


作者实现这个转变，最关键的**逻辑转换**是：**将复杂的、多步骤的“软件流程”，打包成了一个原子化的“硬件指令”**。

-   **以前的做法（软件控制）**：当你需要在加速器上跑一个卷积，代码可能是几千行，包含：配置DMA、启动DMA、等待DMA完成、配置加速器、启动加速器、等待加速器完成、读取结果。每一步都需要CPU参与。
-   **现在的做法**：作者定义了新的RISC-V指令。比如，一个自定义指令 `XCFG CU_ID, SRC_ADDR, LEN`。当CPU执行这条指令时，它**不是自己去做数据搬运**，而是通过XIF接口，**把这条指令“丢”给了协处理器**。
-   **协处理器内部的“大脑”**：协处理器内部有一个**指令控制器**。它接收这条自定义指令后，会自动解码，并管理后续所有步骤：通知CU开始、控制内部互联网络访问SPM、处理握手信号。

你会发现，整个流程中最“重”的那部分——**数据在加速器和本地内存（SPM）之间的流动**——完全由协处理器内部的**硬件流水线**接管了。CPU 只需要发出一条**极轻量级的指令**，然后就可以继续做其他事情。最终，**CPU 的角色从一个具体操作的执行者，转变成了一个宏观任务的调度者**。它不再关心“怎么搬数据”，只关心“哪个数据块要给谁处理”。这个思路，是解决不同时钟域、不同计算单元协同工作难题的通用妙招。

### 2. Multi-bank Scratchpad Memory with LSU and DTCM Mapping

之前的做法为什么“难受”？核心在于**数据搬运的脱节**。

- **高额通信代价**：每次要把数据从Core的主存搬到加速器的本地SPM，都需要通过XIF接口的Custom Instruction。每搬运一个32-bit字，就要**额外消耗5个时钟周期**。这5个周期里，Core的Pipeline是**停滞**的——Core必须充当“搬运工”，干等。
- **软件极不友好**：SPM在Core的**地址空间之外**，意味着Core无法用普通的`load`/`store`指令直接访问加速器的本地数据。程序员必须用`XLOADC`/`XSTORE`等专用指令，手动编排两步流程：先把数据从主存搬到一个“中转站”（Core的Data Memory），再通过Custom Instruction搬到SPM。这不仅冗余，而且**破坏了代码的通用性**。
- **并行度受限**：Core在搬运数据时无法做其他计算，加速器也只能等待数据到位才能启动。整个系统的**吞吐率被Core的搬运速度卡住**。


想象一个**大型工厂**（加速器）和一个**办公室**（RISC-V Core）之间需要频繁交换零件（数据）。

- **旧方案**：办公室先自己把零件从仓库搬到自己的办公桌上（Core主存），然后工人们再通过一个**狭小的文件传票窗口**（XIF），一张一张地把零件规格单递给工厂。工厂拿到单子才能去本地仓库（SPM）找零件。每次传一张单子，窗口都要确认一次，办公室的文书工作完全停顿。
- **新方案**：直接在办公室和工厂之间**开一扇直达的门**（DTCM端口），工厂的本地仓库（SPM）直接变成了办公室的**内部分区**。办公室员工可以随时走进仓库拿东西，就像拿自己办公桌上的文件夹一样。工人（加速器）也从窗口递单子，变成了直接从仓库货架上取货——**两边互不干扰，而且效率翻倍**。


作者并没有大规模重构SPM或加速器，而是**巧妙地“嫁接”了一个连接点**。

- **替换对象**：之前的LSU只能通过XIF的Memory接口与Core通信。作者将LSU**扩展**出一个新的数据通路，直接连接到Core的**Data Bridge**（数据桥）的**DTCM端口**。
- **逻辑转换**：Data Bridge原本负责将Core的原生Data Port协议转换为AXI-4总线，并支持**紧耦合本地内存（DTCM）**。现在，作者将多Bank的SPM**伪装成**一组紧耦合的DTCM内存，通过LSU的另一个接口挂在Data Bridge上。
- **效果**：这样一来，在Core的视角里，SPM的每个Bank都变成了**可寻址的地址范围**（基地址`0x4000_0000`）。Core可以直接用普通的`load`/`store`指令访问SPM，**完全绕过了XIF**。LSU也不再需要驱动XIF的Memory通道，而是直接通过Data Bridge与Core交互。这相当于**把加速器的“本地存储”变成了Core的“一级缓存”**，既消除了Custom Instruction的开销，又让Core能随时读取加速器结果，而加速器也在自己吃自己的数据。

这个设计的精妙之处在于：**没有改动加速器IP本身**，只改动了通信链路的一端（LSU）和映射关系，就实现了从“间接搬运”到“直接访问”的跃迁。

### 3. Scalable Single-Cycle Interconnect Network

*   **传统方案的“两难”**：在异构SoC里，让多个加速器（CUs）和多个内存块（Banks）高效通信是个老问题。
    - 用**全交叉开关**（Crossbar）：连接直觉好，延迟低，但资源开销随端口数平方增长，在FPGA上撑死也就做到8×8，再大就吃光逻辑。
    - 用**共享总线**：资源省了，但一次只允许一对主从通信，多个CU并行时必然串行化，延迟直接爆炸。
-   核心矛盾在于：当CU数量超过4个时，要么**烧资源**（Crossbar），要么**等死**（Bus）。对于需要**多流并发、低延迟**的流式加速器集群，这两种方案都不够实用。


*   **“硬件版二分查找路由”**：想象你要在一条大街上给多位客人递信，但每家每户的门牌号是顺序排列的。
    - 传统Crossbar像是给每位客人配一个**专用投递员**，每个投递员都知道所有门牌号，但你需要很多投递员（资源爆炸）。
    - 传统Bus像是只有一个**共享投递员**，他得按顺序跑完所有门牌，一次只服务一个人（延迟爆炸）。
-   本文的Interconnect就像在街口设置了一棵**“决策树”**：每个路口只看地址的某几位（比如高位），告诉你是向左拐还是向右拐。所有请求同时步入这棵树，每一步都在单时钟周期内完成，最终在叶子节点（内存库）汇合。这棵树的结构非常规整，**不需要每个节点知道全部地址，只需要知道“路径抉择”**，因此逻辑极简，能轻松扩展到32×32甚至更大，而延迟几乎不变。


*   **作者没有另起炉灶设计新协议，而是巧用“对数地址解码”替代“全地址矩阵”**。
    - 他们把地址划分成两部分：**低位**作为内存库的内部地址，**高位**作为库选择信号。
    - 在硬件上实现了**两级树形网络**：一路是**路由树（Router Tree）**（从主设备侧生长），另一路是**仲裁树（Arbitration Tree）**（从从设备侧生长）。两棵树之间通过简单的组合逻辑交换机相连，路径在单周期内建立。
    - 路由树的每个节点只检查地址高位中的**1比特**（对应于树的层级），**“见0向左，见1向右”**，从而以极小的逻辑门数实现路径选择。
    - 仲裁树使用**Round-Robin仲裁**，当多个请求争抢同一个内存库时，轮流服务，避免饿死，且仲裁逻辑也是纯组合电路。
    - 关键效果：**任意主设备到任意从设备的访问延迟恒定为log₂(从设备数)级的组合逻辑延迟**，而不是线性增长。在FPGA上，8×8配置下能跑到333 MHz，资源消耗仅为LUT的1.9%——这比任何Crossbar方案都轻量一个数量级。

*   **为什么能单周期？** 因为整个网络是**纯组合逻辑**（无内部寄存器），地址从CU发出后，直接穿透路由树和仲裁树，在同一个时钟沿到达内存库的地址端口；内存库的响应也通过相同的树反向返回。路径中的所有组合延迟加起来正好在一个时钟周期内完成。这相当于把**传统的多周期总线握手变成了组合逻辑多路复用**。

### 4. Compute Unit with Address Queue for Non-Sequential Access

**痛点直击**

- 传统的流式加速器（AXI4-Stream IP）只能吃下**连续的内存块**——输入地址必须递增，每次+4字节。  
- 但实际应用中，IP 经常需要“跳着读”：比如卷积的滑动窗口、矩阵转置、稀疏索引查找。  
- 之前的做法是：让 RISC-V 核心逐个计算地址，再通过自定义指令逐笔搬运。这导致核心被**粘死**，每个字至少浪费 5 个周期，而且完全无法利用流式 IP 的吞吐潜力。  
- 更糟糕的是，一旦地址模式复杂（如随机步长），软件需要反复执行 `while` 循环下发指令，IP 大部分时间都在空等数据。

---

**通俗比方**

想象你有一个全自动的**烤串流水线**：
- 老方法：烤串机只能按顺序从传送带上取肉串（0号、1号、2号…）。如果你需要先取第5串，再取第1串，再取第8串，你必须自己跑到传送带旁，**手动**把肉串捡出来，再**手动**塞给烤串机。每次只捡一串，烤串机就得等你一次。  
- 新方法：你在流水线旁边挂一个 **“取货清单”**（地址队列），把要取的串号提前写好（5→1→8…）。流水线自己按清单逐个从传送带抓取，然后自动喂给烤串机。你（核心）写完清单就可以去忙别的事，不用再盯着每一个动作。

---

**关键一招**

- 作者并没有修改 AXI4-Stream IP 本身，而是在 **Compute Unit (CU)** 内部增加了一个**地址队列**和一个**数据路径选择器**（见 Figure 6）。  
- **逻辑转换**：将“按地址读数据”拆成**两阶段流水**：
  - **第一阶段（地址加载）**：CU 从 SPM 中连续读取一串地址值（预先存好的“索引数组”），写入内部 FIFO（地址队列）。此时 CU 的读端口被切换到队列方向，数据不进入 IP。  
  - **第二阶段（执行流送）**：CU 从地址队列中弹出值作为 SPM 读地址，返回的数据直接送入 AXI4-Stream IP 的输入端口。地址生成从**硬件计数器**切换为**队列输出**，而 IP 看到的依然是连续的数据流——它根本不知道地址是跳着的。  
- 作者用两条新增自定义指令（`XLDA`、`XCFGA`）控制这两个阶段：`XLDA` 将地址序列加载到队列，`XCFGA` 通知 CU 从队列读取地址进行数据流送。核心只负责下发这两个配置指令，然后释放。  
- 于是，任意访问模式变成了**一次预配置 + 一次自动流送**，不再需要核心逐地址干预。IP 的吞吐几乎不受影响，因为 FIFO 吸收了地址跳转的延迟。

### 5. Forwarding Unit for Direct CU-to-CU Streaming

- 之前的 CU（计算单元）之间要传递数据，必须走一个 **“写入内存 -> 等待完成 -> 再从内存读出”** 的流程。
- 这带来了两重代价：
  - **延迟浪费**：第二个 CU 必须等第一个 CU 把**全部结果**写回内存，才能开始读取，流水线被迫顺序执行，无法重叠。
  - **带宽浪费**：数据先写一次、再读一次，相当于绕了远路，对本地多 bank 的 Scratchpad Memory (SPM) 造成不必要的流量压力。
- 更关键的是，这种做法需要软件通过轮询（`XCFG` 指令反复检查状态）来同步，增加了编程复杂度和控制开销。


把 CU 想象成工厂流水线上的两个工位。  
- **老办法**：工位 A 加工完零件后，必须把零件放到**中央仓库**（SPM），然后工位 B 再去仓库取回来才能继续加工。零件在仓库里进进出出，既浪费时间，又占用了仓库的货架和叉车（内存带宽）。  
- **FWU 的做法**：在 A 和 B 之间加了一条**短传送带**。A 做完一个零件，直接放到传送带上，B 只要伸手就能拿到，根本不用经过仓库。这样 A 和 B 可以**同时工作**，传送带上的零件流动完全由硬件握手控制，流水线无缝衔接。


作者并没有改动 CU 本身的流接口，也没有增加复杂的仲裁逻辑，而是**在从属树（Slave Tree）中植入了一个特殊从属单元：FWU**。

- **逻辑转换**：FWU 本质上是一个**双端口缓冲器**，它被映射为一段“伪内存地址”。  
  - 生产者 CU 对这个地址执行**写操作**（`xcfg`+`xcmp` 配置输出到 FWU 地址）；  
  - 消费者 CU 对这个地址执行**读操作**（配置输入从 FWU 地址来）。  
- **硬件透明**：当两边都就绪时，FWU 在内部直接将写端口的数据**转发**到读端口，**不经过任何内存 Bank**。它只做一件事：在写端准备好时，把数据塞进小型 FIFO，消费端一请求就弹出。  
- **去同步**：由于数据是实时传递的，两个 CU 可以**同时启动**，生产者输出一个字，消费者立刻就能收到，无需软件等待完整批次。这在效果上类似于 CPU 中的**寄存器转发**（Register Bypassing），但作用于流式加速器之间的数据链。

**核心突破**：用硬件缓冲代替了“写内存-读内存”的冗余两步操作，让流式加速器链从**串行**变成了**真正的流水线**。

### 6. Dual-Clock Domain Decoupling

好的，我们来看这个“Dual-Clock Domain Decoupling”。这听起来像个硬件术语，但其实它的核心思想，在现实生活中随处可见。

---


想象一下你的办公室团队：有一位核心领导，能力全面，但他做事慢条斯理，每天只能处理100个文件。而你手下有几个专才，比如一个算数天才，他每秒能算1000个乘法。之前，为了协同工作，你不得不让这个算数天才按照领导的节奏来做事，一秒只能干100个。**这就像让一辆法拉利跟在牛车后面，其强大的加速能力被系统里最慢的那个环节死死拖住。**

*   在之前的架构里，**整个系统（RISC-V核心、协处理器包装器、加速器IP）都必须在同一个时钟频率下运行**。
*   这个频率被**通用RISC-V核心**的瓶颈所限制。加速器IP即使是专门设计来跑高频的，也不得不“降频”工作。
*   这造成了巨大的性能浪费，也让协处理器网络的可扩展性评估失去了意义——因为你测到的瓶颈是核心，而不是你真正想测试的互联网络本身。


这就像一个**城市交通系统**。

*   **RISC-V核心**是城市的**行政区**，负责制定规则和审批文件。它节奏沉稳，时钟周期长。
*   **加速器IPs (CUs)** 是城市里的**高速公路**，需要高频率、快速地跑车。
*   **互联网络**是连接高速公路的**立交桥**。

之前，为了管理方便，整个城市（行政区和高速公路）都使用同一个交通限速——**比如60公里/小时**。这显然不合理，在空旷的高速上开60，太憋屈了。

**“Dual-Clock Domain”的做法，就是在行政区到立交桥的入口处，设置一个“缓冲区”（异步FIFO）。**

*   行政区**以自己60公里/小时的节奏**，把一批车（数据和指令）开到缓冲区门口。
*   缓冲区**按照高速公路120公里/小时的节奏**，把车放进去让它们飞驰。
*   当高速公路上的车跑完了，它们的结果也会先回到缓冲区，然后**再以60公里/小时的节奏**汇报给行政区。

这样，高速公路（加速器）就能跑120码，而行政区（核心）依然在自己的节奏里处理公务，**两者互不干扰，各行其道，各尽其能**。


作者的关键一招，不是一个复杂的电路设计，而是一个**架构上的解耦**。他们没有去改造核心让它跑得更快，也没让加速器跑慢点，而是在两者之间**插入了一个速度转换器——异步FIFO**。

**逻辑转换是：从“全局同步”到“局部隔离”。**

*   **之前是：** 核心和所有加速器被绑在一条绳子上，一荣俱荣，一损俱损。核心是短板，整个系统就得跟着受罪。
*   **作者的做法是：** 将系统清晰地切割成两个“速度域”。一个是 **“慢域”（核心域）** ，另一个是 **“快域”（执行域，包含互联网络和加速器）**。
*   他们用**异步FIFO**作为两个域的连接点。这个FIFO就像一个**水位调节器**，在数据写入时，跟随写入方的时钟（慢域），在数据读出时，跟随读出方的时钟（快域）。它完美地**吸收了两个时钟域之间的速度和相位差异**。

**最终效果：** 这个设计让协处理器的执行单元可以自由地跑出更高的频率（比如表6里展示的625 MHz），而不必再看核心（100 MHz）的脸色。这使得我们能够**真实评估加速器网络本身的性能和可扩展性**，而不是被核心拖后腿。这就像把高速公路和行政区彻底分离，让高速公路测试能真正跑出法拉利的性能。
