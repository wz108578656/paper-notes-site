# Balancing DRAM Locality and Parallelism in Shared Memory CMP Systems 通俗讲解

### 0. 整体创新点通俗解读

- 以前的多核共享内存系统，为了追求**并行度**，把每个线程的请求**均匀打散**到所有DRAM Bank里。这就像把一个团队的成员随机分配到不同房间，大家互相干扰，谁也没法专心干活（失去局部性）。
- 结果就是：虽然并行度很高，但**Row-buffer的命中率**暴跌（比如从98%掉到35%）。每个人都在频繁开新行、关旧行，内存延迟剧增，功耗飙升。

- **真正难受的地方**：你没办法**同时**做到两件事——既要让每个线程的访问尽量“集中”以利用Row-buffer局部性，又要让不同线程的访问尽量“分散”到不同Bank以利用Bank-level并行。以前的做法只能顾一头，另一头就会崩。

---


- 这就像你在一个**共享厨房**里做饭。假设厨房有多个**灶台（Bank）**，每个灶台可以同时烧不同的菜。
- **传统做法**：每个厨师（线程）的食材和锅具被随机分配到所有灶台上。结果就是，厨师A刚在灶台1烧完的锅（Row-buffer），还没刷，厨师B又拿来做别的菜，导致每次都得重新点燃灶台（Precharge + Activate）。效率极低。
- **论文的核心思路**：干脆**给每个厨师分固定的一组灶台**（Bank Partitioning）。厨师A只用1-2号灶，厨师B只用3-4号灶。这样每个厨师都能在自己固定的灶台上连续烧菜，锅不用总刷，烹饪速度（Row-buffer命中率）飙升。
- **但问题来了**：如果厨师A只有2个灶台，而他同时要烧5道菜（内存请求多），他就得排队等灶台，反而更慢（并行度不足）。这时候，作者说：“**别急，我教你一个‘分灶’玩法**”——把每个灶台拆成两个更小的火口（Sub-ranking），这样你原来2个灶台就变成了4个。虽然每个火口火力小一点（访问延迟稍长），但你能同时烧的菜多了，总时间反而缩短了。

- **一句话**：用**物理隔离**保住局部性，用**切分灶台**补偿并行度。就像在厨房里，先画地盘避免串味，再把每个地盘里的灶台改造成更小的独立炉头来提升吞吐。

---


- 作者没有改变DRAM硬件（芯片引脚、指令集），也没有设计更复杂的调度器。他们只做了两件事，却让整个系统“脱胎换骨”：

- **第一招：让操作系统“染色”来分区（Bank Partitioning）**
  - 以前的物理内存分配，是随机或轮询地将物理页帧映射到DRAM Bank。作者改动了操作系统的页帧分配算法，在物理地址中插入一个**颜色位（Color ID）**，确保一个线程的物理页**只落在固定的某几个Bank里**。
  - 巧妙的点：他们没有改内存控制器（地址解码逻辑不变），只是让OS在分配页表时做手脚。这就实现了“软硬件协同”的隔离。

- **第二招：用Sub-ranking来“变出”更多Bank（Sub-ranking）**
  - 传统DRAM的一个Rank是64-bit总线宽，所有芯片同步工作。作者把一个Rank拆成两个独立的32-bit Sub-rank，让它们可以各自独立激活不同Row。
  - 效果：原来一个Bank只能同时保留一个打开的Row；现在每个Sub-rank都可以有自己的Row。这相当于在**不增加物理Bank数量**的情况下，把**有效的并行度翻倍**。虽然每个Sub-rank的请求延迟稍长（因为数据位宽减半），但对于需要并行性的线程来说，这点延迟完全被掩盖了。

- **核心逻辑链**：Bank Partitioning解决局部性 → 导致线程可用Bank变少 → Sub-ranking增加有效Bank数 → 解决并行度不足。两者**互补+协同**，实现了“既要…又要…”的平衡。

### 1. Bank Partitioning via OS-Controlled Page Coloring

- 在现代多核系统中，不同核（线程）的访存流在内存控制器处被交错混合。即使每个程序各自有很好的空间局部性（比如连续访问同一DRAM行），一旦交织在一起，就会互相“踢”对方的行缓冲区，导致频繁的预充电和行激活，大大降低行缓冲区命中率。
- 之前的方案（比如FR-FCFS调度器）只能事后“补救”，靠重排序去恢复部分局部性。但调度窗口有限，请求到来间隔大时，它根本来不及拦下冲突。说白了，这是一场“打完架再劝架”的游戏，效果有限且不治本。


想象一个大型仓库（DRAM），里面有多个货架（Bank），每个货架只有一个临时工作台（Row Buffer）。两个仓库管理员（两个核/线程）各自需要频繁从不同货架上取货。如果没人管，他们可能同时抢同一个工作台，导致双方都无法利用工作台上已经摆好的货物（已打开的行），必须反复把货架推回去再拉出来（预充电+激活），效率极低。

**关键问题：** 工作台有限，管理员们随机抢，没有“地盘”划分。
**本文的做法：** 干脆给每个管理员划定专属货架组。管理员A只允许用1号、2号货架，管理员B只允许用3号、4号货架。这样，A在自己的货架上可以放心地在工作台上堆满货物，B永远不会来捣乱。局部性被完美保留。

- 这就像大学图书馆的自习室，如果大家随机坐，总有人不停抢座位；如果给每个学院分配固定区域，学生就能安心在那里长期占用座位，效率反而更高。


传统的物理帧分配（由操作系统页帧分配器完成）是“颜色无关”的，即物理页帧被随机分配到任意DRAM地址。结果是一个线程散落在所有Bank中，无法避免冲突。

本文的**核心扭转型操作**并非修改硬件（内存控制器不需要改），而是**在操作系统页帧分配算法中植入一个“颜色约束”**：

- 页帧分配器维护一个“颜色”位，该位对应物理地址中用于选择Bank的比特位。  
- 在分配物理帧时，将每个核能够分配的帧限制在**仅有某些特定颜色**的帧范围内。例如，核0只能分配颜色=0的帧，核1只能分配颜色=1的帧。  
- 因为颜色直接解码为Bank索引，所以核0的所有物理帧只会落在Bank 0/1，核1的只落在Bank 2/3。  
- 同时，为了保持cache内集合的均匀分布，他们不是做DRAM层面的XOR置换，而是**置换cache set索引**——这样cache集均匀散列到各bank，但每个核仍然只访问自己的那组bank。

**一句话总结：** 作者没有动一行硬件代码，只是让操作系统在分配物理页时，按照“银行颜色”强行隔离线程的内存空间，从而在根源上杜绝了行缓冲区冲突。后续再用sub-ranking补偿因分区而减少的bank并行度，形成“隔离保局部性，sub-ranking保并行度”的完美互补。

### 2. Cache-Set Permutation to Preserve Bank Partitioning

- 你做了一个非常漂亮的 **bank partitioning**：给每个线程分配专属的DRAM bank集合，彻底隔离行缓冲干扰。
- 但这时候，一个经典优化——**XOR-permuted bank indexing**——跳出来作妖。
  - 这个优化的目的是让同一cache set内的行尽量分散到不同bank，减少bank冲突。
  - 它的做法是：在计算物理地址的bank索引时，**把cache tag的一部分与bank索引做XOR**。
  - 问题来了：这个XOR会**覆盖掉**你辛苦用color（线程ID）定好的bank映射。
  - 结果：线程的页面又被重新打散到**所有bank**，bank partitioning形同虚设。

**所以痛点很清晰**：你既要保持银行隔离（colo固定），又想享受**cache set内行级分散**带来的吞吐增益。这两件事天然矛盾。

---


想象你是个图书管理员，要管理一个大型图书馆（DRAM）。  
- **银行隔离**：你给每个读者（线程）分配**专属的阅览室**（一组bank），确保他们不会互相抢书（行缓冲冲突）。
- **XOR-permuted indexing**：你原本有个规则——每本书（cache line）要放在它所属书架（cache set）的不同分区，避免同一书架的书全堆在一个房间。这个规则是由**书架的编号**（cache set index）和**房间号**（bank索引）做运算决定的。

现在问题来了：  
- 如果你为了让每个读者只在自己房间看书，强行改变房间编号（bank index），那书架编号和房间号的运算规则就**乱了**，读者们会发现书架上的书被自动分到了其他房间——**隔离被破坏**。
- 更好的办法：**不改房间号，只改书架本身的编号方式**。
  - 比如，原来书架编号是1,2,3…排列；现在你用一个**置换函数**重排书架编号（例如1→3, 2→5…）。
  - 这样，同一个房间（bank）里的书架可能来自不同原始书架，但每个读者仍然只拥有特定房间的钥匙。
  - 同时，同一个原始书架（cache set）里的书，因为置换后的编号自然分散到不同房间，**依然能享受行级并行**。

**关键洞察**：冲突的根源是**映射关系**。与其直接动bank索引，不如动**cache set索引**——目标是“以不变应万变”。

---


**作者并没有修改DRAM地址解码逻辑（bank索引的计算方式不变）**，也没有重新设计cache本身。  
他们只做了一件事：**在cache set索引的计算上，插入一个置换（permutation）**。

具体逻辑：

- **原来**：物理地址 → 取中间若干bit作为cache set index → 直接与bank索引关联。
- **现在**：物理地址 → 先通过一个**固定的置换函数**（比如线性变换或XOR）重排这些bit → 得到新的cache set index → 再与bank索引关联。

**效果**：
  - 因为置换是**一对一且可逆**的，同一个color（线程ID）对应的**bank集合保持不变**——隔离没被破坏。
  - 但是，原先属于同一cache set的地址，现在被映射到**不同的新cache set**，从而自然分布到不同bank——**行级分散的效果依然存在**。

用一个极简例子理解（假设只有2个bank，color是bit0）：  
- 原始地址：A(0,0), B(0,1), C(1,0), D(1,1) —— color=bit1（假设bank由bit1决定）。  
- 原来cache set index=bit0：（A,B）在set0，（C,D）在set1。  
- 若XOR bank index和tag（假设tag是bit0），那么A和C的bank会互换——隔离破坏。  
- 现在**置换**cache set index：比如把bit0和bit1交换作为新set索引——于是原set0中的A,B被分到新set{0,1}，分别落在bank0和bank1——同一set内的行自然分散到不同bank。

**一句话总结**：**把搅局者的作用对象从“bank索引”转移到“cache set索引”，从而在保持银行隔离的情况下，仍然实现cache set内行的银行级并行。**

### 3. DRAM Sub-Ranking to Increase Effective Bank Count

传统的 **DRAM 系统**里，一个 **Rank** 就像一条 **64 位宽的超级高速公路**——所有的芯片（Chips）必须步调一致地工作。当你把内存银行（Banks）划分给不同线程（Bank Partitioning）后，每个线程能用的银行数量变少了。这时候，如果有一个线程突发大量并行内存请求，它就会因为**能同时工作的“车道”不够**而堵车。更糟的是，增加物理 Rank 或 Channel 又贵又受引脚限制。所以，**核心矛盾**是：你手头只有有限个“大房间”，却要招待越来越多的“客人”（线程），每个客人还都想同时用好几个“房间门”（Bank）。这就是**银行级并行度（Bank-Level Parallelism）**不足的“难受”之处。

想象你只有一个 **大教室**（一个宽 Rank），里面有 8 个**独立的小格子**（Bank）。平时上课，全班 64 个人（对应 64 位数据）必须一起走进教室，然后所有人只能同时使用同一个“大门”（Rank 的全局控制信号）。如果你把 8 个格子分给 4 个小组（线程），每个小组就只能用到 2 个格子。当一个小组想同时做 4 件事时，另外 2 个格子就闲置了——**因为大门一次只能让一组人进出**。

**Sub-Ranking 的做法**：你把这个大教室拆成 **两个小教室**（32 位宽的子 Rank）。每个小教室都有自己的独立大门和门卫。这样一来，即使每个小组只分到 2 个格子，但因为两个小教室可以同时开工，你**实际上拥有了两倍于之前的“并行门数”**。这就好比在一个大商场里，原本只有一个出入口，现在改成了两个独立出入口，人群分流后，每个人都能更快地进出。

作者并没有增加物理内存条或者改变芯片内部结构，而是**在 DRAM 的控制逻辑上动了一个“脑筋”**。

- **原来的做法**：一个 Rank 内部的所有芯片共享同一个 **Chip Select** 信号，必须同时激活、同时读/写。所以，即使 Bank 是物理独立的，它们的控制时机却是**绑定**的。
- **Sub-Ranking 的手法**：作者将一个宽 Rank（例如 64-bit）拆成多个**更窄的子 Rank**（例如 32-bit 或 16-bit）。每个子 Rank 拥有**独立的 Chip Enable 信号**。这意味着：
  - 当 CPU 发起一次内存访问时，**只有对应的子 Rank 中的芯片被唤醒**，其他子 Rank 可以保持空闲或干别的事。
  - 由于子 Rank 之间互不干扰，**每个子 Rank 内部的所有 Bank 都可以独立地打开不同的 Row**——相当于把原本一个 Rank 的“同步操作”变成了多个子 Rank 的“异步并发”。
- **代价权衡**：拆得越细（比如 16-bit 子 Rank），并行度越高，但每次访问需要读更多次才能凑够原来的数据宽度（因为每次只读取一小部分），导致**延迟增加**。作者选择 **32-bit 子 Rank** 作为平衡点——既获得了足够多的有效 Bank 数，又把额外延迟控制在可接受范围内（大约 40 个 CPU 周期）。

**一句话总结**：作者**并没有改变 DRAM 的物理 Bank 数量，而是通过切断 Rank 内部的“统一控制线”，让原本绑在一起的 Bank 们“松绑”，从而用“分而治之”的思路凭空增加了可用的并行度**。

### 4. Combined Bank Partitioning and Sub-Ranking

在传统的CMP（片内多核）系统里，内存系统过得很憋屈。它有两个相互矛盾的“刚需”，但就像左右手互搏，怎么也协调不好：

- **刚需一：高局部性（Locality）**。为了节能和低延迟，DRAM希望多命中**行缓冲**。这意味着多个访问请求最好能扎堆在同一个“行（Row）”里。这是DRAM的高效工作模式。
- **刚需二：高并行性（Parallelism）**。为了隐藏访问延迟，系统又希望多个请求能同时发往不同的**Bank**，让它们并行工作，免得一个请求卡住整个系统。

**问题出在哪？** 在单核时代，这两个需求是自洽的。但一旦多核共享内存，各个核的访问流就像是没规规矩矩排队的初中生，你推我搡。原本一个核发起的连续访问（高局部性），会被其他核的“捣乱”请求冲散。**局部性好，但搞没了；想靠并行性补救，银行的个数又不够用。**

你想想，银行分区（Bank Partitioning）把每个核的访问流隔离开，保住了局部性，但等于把每个核能用的Bank变少了——**并行性又不够了**。而单纯加子排名（Sub-Ranking），把一个大银行拆成多个小程序块，虽然增加了有效Bank数（提升了并行性），但这种拆分本身会引入额外的访问时长和复杂性，**而且并没有解决核间请求互相干扰、冲走行缓冲的根源问题**。


想象一个**大型书市**，有好几个不同的图书分区（Memory Banks）。每个分区里有很多书架（DRAM Row）。有个勤快的店员（Memory Controller）。

- **传统的做法是**：所有顾客（不同Core的访问流）都挤在一个分区里，同时翻书。A顾客刚看到书架上的《高性能计算》，还没来得及抽出来，B顾客就把书架推走了。店员只好把书架推回去（预充电），再推来另一个书架。**这就是“行缓冲颠簸”**，效率极低。
- **银行分区就像**：给每个顾客划定一个**专属阅读区域**。A顾客只能在1号区看，B顾客只能在2号区看。这样A把书架摊开看多久都行，不会被B打扰。**局部性回来了**。
- **但问题大了**：专属区域太小了。如果A顾客想看10本书，而他的区域里只有2个书架（2个Bank）。他得等一本看完、放回去，才能拿下一本。**并行性极差**，排队排到死。
- **纯子排名就像**：把所有的书架全部拆掉，换成更小、更多的**杂志架**。原先2个大书架，变成了8个小杂志架。并行性增加了，A可以同时翻开多本杂志。**问题是**：新杂志架的尺寸是适合A的喜好，但B的干扰也还在，A想把一小摞同主题的杂志堆在一起看，B一来，又得乱套。

**作者的神来之笔**：不是死板地把“专属区域”和“多杂志架”二选一。而是：先给每个顾客划定了专属的**大书架区**（Bank Partitioning，保证互不干扰），然后**在这个专属区内**，把每个大书架拆成几个小杂志架（Sub-Ranking，增加有效Bank数）。这样，A在自己的区域里，既能不受打扰地看一本书（局部性），又能同时从多个杂志架上拿书（并行性）。


作者做的非常简单且巧妙，并非新造一个复杂的控制器：

- **核心思路**：不是“调度”去避免冲突，而是从根源上“物理隔断”访问流。这个思路借鉴了操作系统的**页面着色（Page Coloring）**。
- **具体的操作**：作者修改了**操作系统的物理帧分配算法**。在分配物理内存页给某个进程时，不是随机分配，而是**强制要求**该进程的物理页面，必须被映射到属于同一个**颜色（Color）** 的DRAM Bank集合里。这个“颜色”就是一组唯一的、不与其他进程共享的Bank ID。
- **为什么这招很“狠”**：它把**软件层的虚拟地址映射**和**硬件层的DRAM Bank布局**直接挂钩了。OS说“这个页给你”，实际上是在告诉内存控制器：“这块数据，必须走到那个特定的Bank里去。” 这就等于给每个核心的访问流分配了**独立的物理车道**。
- **怎么补充并行性**：单车道当然堵车，所以作者同时用**子排名（Sub-Ranking）** 来扩宽车道。标准DRAM里，一次访问必须激活整个Rank（通常8个Bank）。子排名允许只激活**一半的Bank（例如32-bit子排名，只用4个Bank）**。这样做的好处是，虽然每个“车道”变窄了（单次访问带宽变低），但车道数量翻倍。每个核在自己的“颜色集”里，能用的有效Bank数翻倍，从而补偿了因分区而损失的**并行性**。

**一句话总结**：作者甩开调度器的“和稀泥”，直接在地址映射这个起跑线上“划地盘”，再用子排名给每个人的地盘多划几道“虚拟单车道”。这招让硬件负责隔离保障局部性，让软件负责分配资源保障并行性，两者精准互补，像是一道完美的配方。
